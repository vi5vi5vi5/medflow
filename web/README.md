# web/

Фронтенд. Статика, которую раздаёт API-сервер из [`../api`](../api):
он монтирует эту папку на `/` (см. `setupMisc()` в `ApiServer.cpp`)
и плюс несколько «pretty URL» поверх — `/clinic/:id`, `/doctor/:id`,
`/patient/:id` мапятся на соответствующие `.html`.

Никакого build-шага: ни npm, ни webpack, ни TypeScript. Браузер тянет
React и Babel-standalone с CDN, JSX компилируется прямо в рантайме.
Это сознательный выбор: проект учебный, один разработчик, нулевой
порог входа.

## Файлы

| Файл               | Что это |
|--------------------|---------|
| `index.html`       | Вход: выбор роли (клиника / врач / пациент) и переход в нужный дашборд. |
| `clinic.html`      | Дашборд администратора клиники. |
| `doctor.html`      | Дашборд врача (расписание, приёмы). |
| `patient.html`     | Дашборд пациента (запись на приём, история). |
| `clinic-views.jsx` | Общие React-компоненты для clinic-дашборда (вынесено, потому что разрослось). |

## Соглашения

- Шрифты — Inter Tight (sans) и JetBrains Mono (mono) с Google Fonts.
- Цветовая палитра задана CSS-переменными в `:root` каждого `.html` —
  если будешь менять тему, держи `--bg`/`--ink`/`--accent`/... в синхроне
  между файлами.
- Все запросы к API — через `fetch('/api/...')`, без отдельного клиента.
  CORS открыт (см. `ApiServer.cpp::setupCors`), но т.к. фронт ходит с
  того же origin, это не нужно — оставлено на случай отдельного хоста.
- Логика валидации **не дублируется**: фронт показывает удобные подсказки,
  но финальная проверка — на сервере, в [`../services`](../services).
  Ошибки сервиса прилетают как `{"error": "..."}` с 400 и просто
  выводятся пользователю.

## Если фронт не отдаётся

В `setupMisc()` сервер ищет папку `./web` относительно бинаря, потом
`../web`, потом `../../web` — на случай запусков из разных рабочих
директорий. В Docker-образе `web/` копируется в `/app/web` рядом с
бинарём (см. `Dockerfile`), путь `./web` срабатывает первым.
