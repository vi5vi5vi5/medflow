# interface/InMemory/

Реализация репозиториев поверх `std::unordered_map<int, Xxx>`. Никакого
диска, никаких миграций — данные живут только пока процесс работает.

## Когда используется

В `main.cpp` **сейчас не подключена** — там собирается всё через
`Sqlite*` из [`../sqlite`](../sqlite). InMemory остаётся как:

- быстрый стенд для ручных проб (`make_shared<InMemoryPatient>()`
  вместо `make_shared<SqlitePatient>(db)` — и сервис заработает),
- основа для будущих юнит-тестов сервисного слоя без БД.

## Что внутри

По файлу на сущность, все наследуют соответствующий интерфейс из
[`../Abstract`](../Abstract):

- `save`: если `id == -1` — выдаём `nextId_++`, иначе — двигаем `nextId_`
  до `max(id+1, nextId_)` (чтобы импорт с фиксированным id не сломал
  последующую авто-генерацию).
- `findBy(predicate)`: линейный обход. На «больших» данных не
  годится — но в памяти у нас никогда не «большие».
- Связи между сущностями не валидируются: за это отвечает сервисный
  слой, а не репозиторий.

## Отличия от sqlite-реализации

- Нет общего «коннекта» — каждый репозиторий полностью самостоятельный.
- Нет мьютекса: рассчитано на однопоточное использование (тесты, скрипты).
  Если понадобится дёргать из API — оборачивать снаружи или добавлять
  локальный mutex.
