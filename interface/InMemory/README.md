# interface/InMemory/ — репозитории в памяти

Реализация контрактов из [`../Abstract`](../Abstract) поверх
`std::unordered_map<int, Xxx>`. Ни диска, ни схемы, ни миграций — данные живут,
пока жив процесс. По файлу на сущность, каждый класс наследует свой `IXxx`.

## Когда используется

В [`../../MedFlow_BY_LLl-23.cpp`](../../MedFlow_BY_LLl-23.cpp) **сейчас не
подключена** — там всё собрано на `Sqlite*` из [`../sqlite`](../sqlite). InMemory
остаётся как:

- быстрый стенд для ручных проб — заменить `make_shared<SqlitePatient>(db)` на
  `make_shared<InMemoryPatient>()`, и сервис заработает без БД;
- основа для будущих юнит-тестов сервисного слоя.

## Что внутри

Каждый репозиторий держит свою `unordered_map` и счётчик `int nextId_ = 0`.

### Генерация id (`save`)

```cpp
if (obj.id == -1) obj.id = nextId_++;          // новый — выдать следующий id
else if (obj.id >= nextId_) nextId_ = obj.id + 1;  // импорт с готовым id — сдвинуть счётчик
map_[obj.id] = obj;
```

Вторая ветка нужна, чтобы импорт записей с фиксированными id (из бэкапа) не
приводил потом к коллизии при авто-генерации. Соответствует upsert-контракту из
[`../Abstract`](../Abstract).

### Поиск

- `findById` — прямой `map.find`, O(1).
- `findBy(predicate)` — линейный обход всей мапы. На «больших» данных не годится,
  но в памяти у нас никогда не «большие».
- `ISpecialization::findBy(name)` — обход с поиском подстроки в `name`, возвращает
  первое совпадение (как и sqlite-версия).

### Чего здесь нет

- **Валидации связей** между сущностями — это забота сервисов, не репозитория.
- **Порядка** — `unordered_map` не упорядочена; `findAll()` отдаёт записи в
  произвольном порядке.

## Отличия от sqlite-реализации

| | InMemory | [sqlite](../sqlite) |
|---|---|---|
| Хранилище | `unordered_map` в RAM | файл `medflow.db` |
| Общий «коннект» | нет, каждый репозиторий сам по себе | один `SqliteDb` на всех |
| Потокобезопасность | **нет** мьютекса — только однопоточно (тесты, скрипты) | `lock_guard` на каждой операции |
| Переживает рестарт | нет | да |

Если вдруг понадобится дёргать InMemory из многопоточного API — оборачивать
синхронизацией снаружи или добавлять локальный `mutex`.
