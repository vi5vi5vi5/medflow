# interface/

Слой персистентности: абстрактные репозитории и их реализации.

```
interface/
├── Abstract/    интерфейсы (чистые virtual) — то, на что опирается сервисный слой
├── InMemory/    хранение в std::unordered_map — для тестов и быстрых проб
└── sqlite/      боевая реализация поверх SQLite (используется в main)
```

## Зачем разделение

Сервисы из [`../services`](../services) принимают `shared_ptr<IPatient>`,
`shared_ptr<IClinic>` и т.д. — никогда конкретный `Sqlite*` или
`InMemory*`. Это значит:

- В `main.cpp` собирается сборка из конкретных реализаций (сейчас —
  все `Sqlite*`, через общий `SqliteDb`).
- В тестах или в одноразовых утилитах можно подсунуть `InMemory*` —
  без диска, без миграций.
- Если когда-нибудь понадобится Postgres / sqlite через ORM /
  что-то ещё — добавляется ещё одна папка-сосед, без правок в сервисах.

## Что где смотреть

- Сигнатуры репозиториев и доменные «контракты» (`save`, `findBy`,
  `removeBy`, и т.п.) — в [`Abstract/`](Abstract).
- Реальные SQL и схема таблиц — в [`sqlite/SqliteDb.h`](sqlite/SqliteDb.h)
  (один общий коннект, мьютекс на операцию, WAL).
- Если интересно, как сущности живут «без БД» (id-генерация, поиск
  по предикату) — [`InMemory/`](InMemory).
